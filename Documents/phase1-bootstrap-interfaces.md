# 1단계: Bootstrap + 인터페이스 정의

## 수행 내용

### 1.1 폴더 구조 생성
```
Assets/Scripts/
├── Bootstrap/          # DI 컨테이너
├── Core/
│   ├── Interfaces/    # 5개 인터페이스
│   └── BaseClasses/    # 2개 베이스 클래스
├── Pool/               # 오브젝트 풀링
└── Game/               # 게임 매니저
```

### 1.2 인터페이스 정의 (5개)
| 인터페이스 | 용도 |
|------------|------|
| IPlayerController | 플레이어 이동 조작 |
| ITerrainGenerator | 레인 생성/관리 |
| IObstacle | 장애물 기본 계약 |
| IGameManager | 게임 상태/점수 |
| IObjectPool | 제네릭 풀링 |

### 1.3 베이스 클래스 (2개)
| 클래스 | 용도 |
|--------|------|
| BaseLane | 레인 공통 기능 (위치, 타입) |
| BaseObstacle | 장애물 공통 기능 (활성화/비활성화/이동) |

### 1.4 Bootstrap (DI 컨테이너)
- **역할**: 의존성 주입 관리, 게임 초기화
- **핵심 기능**:
  - `RegisterDependency<T>()`: 인터페이스에 구현체 등록
  - `Resolve<T>()`: 등록된 의존성 해결
  - 오브젝트 풀 2개 관리 (Lane, Obstacle)

### 1.5 ObjectPool<T> (제네릭 풀링)
- **역할**: 가비지 컬렉션 최소화, 오브젝트 재사용
- **공비**: `Stack<T>` (가용) + `HashSet<T>` (대여 중)
- **메서드**: `Rent()`, `Return()`, `ReturnAll()`, `Prewarm()`

## 왜 이렇게 설계했는가?

### SOLID 원칙 적용
| 원칙 | 적용 근거 |
|------|-----------|
| **I** (인터페이스 분리) | 5개 인터페이스로 책임 분산 |
| **D** (의존성 역전) | Bootstrap에서 인터페이스에 주입 |
| **S** (단일 책임) | 각 클래스는 단일 역할 |

### 오브젝트 풀링
- **이유**: 엔드리스 러너는 레인/장애물이 빠르게 교체됨
- **효과**: 매 프레임 new/delete 방지 → GC 부하 감소

### DI 컨테이너
- **이유**: 플레이어 ↔ 지형 ↔ 장애물 간 느슨한 결합
- **효과**: 테스트 용이, 구현체 교체 자유로움
